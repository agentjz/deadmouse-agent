# Kitty 哲学

## 历史与当下

过去应该怎样进入现在？

如果旧任务、旧判断和旧惯性被整段塞进当下，它们会伪装成当前意图，把模型拖回已经过去的任务。

如果历史被一刀切地隔离，模型又会像每句话都重新开局：用户刚刚确认过的事情要重复解释，刚刚走过的路要重新读取，长任务会变笨，token 会被浪费在反复找回上下文上。

Kitty 当前的选择不是回到旧历史，也不是清空桌面，而是把上下文分层：

- 同一个 session 的近期对话脉络会被结构化压缩后进入当前轮，负责用户连续体验。
- 当前任务工作记忆自动进入当前轮，负责执行连续性。
- 完整历史、最终输出、工具产物、运行事件和变更记录留在证据库里，只在需要取证时查询。
- 当前上下文是模型当下工作的桌面，只放当前用户输入、同 session 对话脉络、当前任务工作记忆和最小必要工作集。
- 长期记忆暂不产品化。项目长期原则应该写进可审阅的项目文档、测试和源码，而不是沉到不可见的用户画像里。

记忆不是聊天记录，历史也不是上下文。记忆负责续命，历史负责取证，上下文负责当下推理。

这条边界同时保护两件事：用户不会感觉每一轮都断片，模型也不会被旧任务污染当前目标。

## 用户与事实

Agent 应该更忠于用户，还是更忠于事实？

用户友好可以降低摩擦，也可能让用户更舒服地停留在自己的旧习惯、旧偏好和旧盲区里。

Kitty 当前的选择是：理解用户，但不讨好用户；尊重偏好，但不让偏好越过事实。用户的上下文可以被接住，用户的历史可以被查询，但事实证据仍然是边界。

## 核心与工具

能力越多，Agent 是否就越强？

不一定。默认能力越多，模型越容易分心，用户也越难判断系统到底在做什么。

Kitty 当前的选择是：核心先只保留 `read / edit / write / bash`。复杂能力以后可以作为明确扩展重新进入，但不能污染默认核心。

## 模型与机器

判断应该交给模型，还是交给机器？

机器更可靠，模型更会判断。机器如果开始替模型决定策略，会变成第二个大脑；模型如果只靠语言声称完成，又会失去事实地面。

Kitty 当前的选择是：模型负责活判断，机器负责死事实。模型决定路线，机器记录、执行、保存证据、维护账本、等待变化和校验硬边界。
