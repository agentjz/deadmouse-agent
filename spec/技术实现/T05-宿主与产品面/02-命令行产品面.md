# 命令行产品面

## 文档目的

本文说明 CLI 当前的产品面结构，以及 fast path、lightweight path 和 full runtime path 的分工。

## 模块目标与当前状态

CLI 当前负责把常用命令分层启动，避免轻命令拉起整套 runtime，同时保证 one-shot 和 interactive 仍走统一宿主边界。

## 正式边界

主要代码边界：

- `src/cli.ts`
- `src/cli/program.ts`

- `src/ui/todoStyling.ts`

## 主路径

当前 CLI 启动链分为三类：

- fast path：`--version`、`help`、轻量信息命令
- lightweight path：不需要完整 runtime 的配置与诊断命令
- full runtime path：one-shot、interactive、宿主服务等重路径

## 模块职责与非职责

### CLI 产品面负责

- 按命令类别组织启动成本
- 暴露产品级命令入口
- 在需要时转入统一 runtime 或 host 路径
- 通过 runtime-ui 统一运行输出策略（来源切换标题、工具语义标签、execution 回执、工具失败收据、todo 标记着色）

### CLI 产品面不负责

- 定义核心真相源
- 绕开 `src/host/` 直接执行业务 turn
- 继续长成单个总控大文件

## 当前落地决定

CLI 当前已经固定为“轻命令快速返回，重命令走正式 runtime”的产品面结构。
当前运行输出固定先进入 `RuntimeUiEvent`，再由 shell sink 渲染为终端文本。
Lead、Dreaming、workflow、subagent、team、background 和 system 是运行来源 channel，但终端不在每行重复来源前缀。
来源只在切换时输出一次中文 section header，例如 `[决策主脑]`、`[做梦]`、`[工作流]`、`[子代理]`、`[队友]`、`[后台]`、`[系统]`，连续同源输出不重复盖章。
身份显示名由 runtime-ui 的集中映射表维护，不允许散落在渲染逻辑里硬编码。
每行主视觉保留工具语义标签：`[tool]` 表示工具调用，`[dispatch]` 表示派发，`[result]` 表示结果，`[preview]` 只承载允许展示的预览。
工具调用、工具失败、前台 execution stream、dispatch 和状态消息都走 runtime-ui 的统一摘要、颜色和 preview 策略。
工具成功默认静默，不输出 `[result] ... ok`；只有失败输出红色 `[result] ... failed`。
终端强调色收敛为三类：身份 section header 与失败使用红色，工具动作与当前 todo 使用紫色，非 Lead 来源正文、reasoning 和弱信息使用暗色。
非 `todo_write` 不展开大块 preview；失败原因压到同一行；`todo_write` 预览保留 `[ ]/[>]/[x]` 语义并在终端展示，因为它是任务仪表盘，不是原始证据。完整工具结果继续交给工具消息、落盘结果和 tracked 证据链承载。
