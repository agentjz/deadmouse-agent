# Multi-Agent Scheduling

## Document Purpose

This document describes the current execution lane model: how Lead coordinates protocol-declared execution work, and how runtime state is recorded.

## Current Model

Deadmouse uses one execution lifecycle with two lanes:

- `agent` lane
- `command` lane

Profiles:

- `subagent` and `teammate` are `agent` profiles.
- `background` is a `command` profile.
- `workflow` is an `agent` profile.
- `dreaming` is an `agent` profile for external agent ecology execution.

## Truth Source And State Ownership

Execution state is owned by execution records. Primary statuses are:

- `queued`
- `running`
- `paused`
- `completed`
- `failed`
- `aborted`

Lead waits on execution records whose `deadmouse.lead-wait-policy` snapshot declares `while_execution_active`. It does not own a private polling loop and it does not spend model turns checking execution status while nothing changed.

Dreaming executions add a foreground stream while Lead is suspended.
The stream is observable terminal output for the active external ecology; the truth source remains execution closeout and recorded artifacts.

## Runtime Story

1. Lead decides whether to start an execution, which capability to use, names, roles, and assignment text.
2. The model calls an explicit tool; the machine validates deterministic invariants and creates worker-backed execution records through the shared assignment execution path.
3. Workers start through the shared `__worker__ run --execution-id <id>` entry.
4. Each execution records the `AssignmentContract`, `CapabilityPackage`, `deadmouse.execution-policy`, and `deadmouse.lead-wait-policy` snapshots used at creation time.
5. Each `CapabilityPackage` snapshot includes `deadmouse.capability-port`, which declares the execution dock, permission boundary, foreground output, artifact expectations, closeout behavior, and wake behavior.
6. Each execution receives `deadmouse.execution-boundary` with runtime and idle limits.
7. Agent and command executions run until completion, real failure, user interruption, or execution boundary.
8. Boundary hits close the execution as `paused` with a structured reason and return control to Lead review.
9. Normal completion and failure also close through `closeExecution(...)`.
10. Dreaming executions additionally run inside a Mirror World worktree, stream progress to the foreground, write a merge proposal, and verify that Real World project files were not changed.
11. Execution closeout publishes an event and sends a message to Lead inbox.
12. Lead wakes, reads execution closeout and related facts, then decides the next move.
13. Wake signals are not truth sources; closeout records are.

## Responsibilities

Execution lane owns:

- durable execution status
- worker launch protocol
- runtime and idle boundaries
- foreground execution stream references for profiles that declare them
- closeout records
- closeout events

Lead owns:

- delegation judgment
- merge/review decisions after closeout
- next-move judgment

Subagent/team workers own:

- assigned execution work
- evidence collection
- closeout summary and artifacts

Dreaming runner owns:

- Mirror World creation
- Mirror World-only write boundary
- foreground stream emission
- merge proposal artifact
- Real World unchanged verification

## Non-Responsibilities

Execution lane must not:

- decide task strategy for the model
- invent delegation without Lead tool calls
- create task-board stages from prompts, wake signals, checkpoints, or compaction
- stop subagents because of low tool-call or model-turn budgets
- treat wake signals as state truth
- merge Dreaming results into Real World without explicit user approval

Lead must not:

- self-poll execution work with empty model turns
- declare completion without closeout evidence
