# 本地命令与流程

## 文档目的

本文记录仓库当前使用的本地命令和标准验证流程。

## 当前核心命令

- `npm.cmd run sync`
- `npm.cmd run sync:readme-capabilities`
- `npm.cmd run verify:generated`
- `npm.cmd run verify:repo-contracts`
- `npm.cmd run live:ecology`
- `npm.cmd run check`
- `npm.cmd test`
- `npm.cmd run audit:tests`
- `npm.cmd run build`
- `node dist\\cli.js`

## 测试分层

仓库测试分为两套逻辑：

- 普通测试：`npm.cmd test`。它验证源码、类型、构建、单元行为、协议边界和仓库契约，属于每次开发收口必须跑的本地质量线。
- Live API 实战测试：`npm.cmd run live:ecology`。它只在项目所有者明确要求真实 API 体检时运行，不进入 `npm.cmd test`、不进入 `npm.cmd run verify`。

Live API 实战测试的目标不是替代普通测试，而是验证真实模型带着 Kitty harness 能否真刀实枪调用工具、能力、工作流和生态闭环。

Live API 实战测试必须：

- 在独立 Mirror World 中复制、安装、构建并运行，不能直接在 Real World 源码目录里执行模型写入。
- 覆盖文件、代码、shell、task、worktree、background、Dreaming、Dreaming Loop、subagent、team、workflow、history、trace、network、capability package 等能力面。
- 对外部生态允许显式 skip，例如 MinerU 可以在缺少外部条件时跳过，但 skip 必须写入报告。
- 记录每个工具的调用证据、失败原文摘要、缺失项和输出目录。
- 跟随能力生态同步更新；`spec/用户审阅/capability-ecology.json` 是 README 能力表和 Live API 工具开关的共同事实源，每个工具必须声明 live group、enabled/disabled 和 disabled 原因。

## 当前标准流程

1. 先确认本次变化属于实现、协议、生成物、文档还是测试结构。
2. 涉及生成物时先改事实源，再跑 `npm.cmd run sync`。
3. 行为变化先补失败测试或边界测试；测试粒度保护边界，不锁死无关细节。
4. 修改实现和 spec。
5. 跑 `npm.cmd run verify:generated`、`npm.cmd run verify:repo-contracts` 和相关测试。
6. 收口前跑 `npm.cmd test`。

## 当前约束

- Windows 环境优先使用 `npm.cmd`
- `verify` 必须保持标准入口：`npm run verify:repo-contracts && npm test`
- `scripts/` 只保留项目所有者手动使用的快捷脚本；自动化生产线脚本放在 `tests/production-line/`
- `tests/production-line/` 是统一生产线层；共享运行、工具注册、证据和参数解析逻辑必须集中治理，单个脚本只表达测试场景和验收断言
- Node 自动化生产线脚本必须使用 TypeScript，并进入 `npm.cmd run typecheck`
- `verify:repo-contracts` 会检查生成物漂移，不能绕过 `sync`
- `verify:repo-contracts` 会检查 Live API 生态库存是否覆盖所有已注册工具，不能维护第二份工具开关清单
- `verify:repo-contracts` 会阻止拆字、base64、fromCharCode 等伪装字符串拼贴；旧残留检查不能靠隐藏字面量绕规则
- 涉及用户入口时，优先验证 CLI、Telegram 的正式主路径
- 涉及真相源、Lead/机器边界或宿主边界时，不只看单元测试，还要看对应集成路径
- `audit:tests` 是审计，不是门禁；它帮助判断测试是否过胖或过细
