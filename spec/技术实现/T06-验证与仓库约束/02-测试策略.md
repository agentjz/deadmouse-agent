# 测试策略

## 文档目的

本文定义仓库当前的测试目标、测试分层和收口标准。

## 当前优先保护对象

1. Lead 的耐跑与恢复能力
2. Lead 显式派发、机器等待和回主回合语义
3. 会话现场、控制面账本、trace 和配置入口
4. 宿主运行边界
5. 工具协议、扩展边界和 capability package 治理
6. provider 适配与脚本化 provider harness
7. regression case 的证据捕获和运行

## 当前测试分层

### 1. 结构护栏

用于防止架构回退，例如：

- 宿主不得直连底层 turn
- 宿主不得直连 raw runtime registry
- 关键目录与关键文件保持职责清晰
- protocol、capabilities、agent、host、observability、trace、regression 不互相吞并职责

### 2. 真相源回归

用于保护正式状态边界，例如：

- session snapshot
- checkpoint / verification / acceptance
- trace dossier
- SQLite control plane
- capability package manifest
- regression case file
- 配置入口

### 3. 行为回归

用于保护用户可见能力，例如：

- one-shot closeout
- interactive 行为
- Telegram `/stop`
- 可见输出与文件回传
- 工具使用主路径
- capability package CLI
- regression CLI

### 4. 链路验证

用于证明关键模块在真实主路径中可工作，例如：

- provider 接入链路
- scripted provider harness
- host turn
- execution lifecycle
- observability 旁路记录
- trace 取证记录

## Regression Case

regression case 是从已发生 session 和 trace 中捕获的证据用例。

它的职责是：

- 固化 session id、turn id、trace event kind、最终 assistant 文本和事件数量等事实
- 在后续运行中检查这些事实是否仍可被读取和满足
- 帮助定位是现场、trace、provider、工具还是执行边界发生退化

它不是：

- 新的策略引擎
- 模型自动评审器
- 替 Lead 判断完成度的机器层
- 对真实 provider 的替代

## 当前规则

1. 行为变化时测试必须先行或同步更新。
2. 模块边界变化时，测试要同步迁移。
3. 文档驱动和测试驱动必须同步工作。
4. 真相源、Lead/机器边界、宿主边界、trace、regression case 和扩展治理的改动必须有对应保护测试。
5. scripted provider harness 用于离线复现 provider 行为；真实 provider 能力仍由 adapter、doctor 和集成链路验证。

## 当前落地决定

当前仓库把测试视为结构护栏，不把测试当覆盖率装饰。判断是否收口，优先看主路径、正式真相源和回归证据是否被保护，而不是只看单点断言是否通过。
