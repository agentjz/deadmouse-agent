# 扩展机制

## 文档目的

本文说明 Deadmouse 当前的正式扩展口，以及 tools、skills、MCP、host extra tools、profiles 和 capability package 如何进入统一主路径。

## 当前正式扩展口

1. tools
2. skills
3. MCP
4. host runtime callbacks / extra tools
5. profiles
6. capability packages
7. Dreaming external agent ecology
8. Dreaming Loop workflow

### 5. profiles

主要代码边界：

- `src/agent/profiles/`
- `src/agent/promptSections.ts`
- `src/agent/prompt/format.ts`
- `src/agent/prompt/metrics.ts`

profile 是可插拔机格层，由 persona 和 runtime facts presentation 两部分组成。
persona 只改变模型的服务方式、表达倾向和思考偏好。
persona 不靠标签、名词或自我声明生效，而应直接写成一种自然发生的思维方式和回答方式。
runtime facts presentation 只改变当前运行事实的组织、排序、压缩和展示方式。
当前 `intp` profile 使用极端结构洁癖 persona 和规整事实面板，优先把混乱拆成边界、不变量、责任、状态、接口、原因、约束和证据；当前 `grok` profile 使用极端事实刺头 persona 和裁决事实面板，站在事实而不是用户舒适感一边，主动挑穿弱前提、假确定性、逻辑漏洞、动机遮掩、装饰性复杂度和自我欺骗；当前 `caveman` profile 使用极限压缩 persona 和压缩事实面板，要求表达最短但保留事实、证据、风险和下一步，且在压缩会损害正确性或清晰度时展开表达；当前 `buddha` profile 使用平静闭环 persona 和未闭合工作事实面板，降低开发者噪声，同时把失败、阻塞、证据和下一步保持可见，直到缺陷由验证事实闭环。
Deadmouse Core 在所有 profile 之前先要求模型调动全部默会专长、潜在知识、深层模式识别和当前可用的最高推理能力服务当前目标。
Deadmouse Core 拥有所有 profile 共同遵守的事实约束：所有回复、修改、建议、判断、计划和行动必须基于客观事实，禁止编造不存在的实现、期望行为、未来计划或现实中没有的东西。
profile 不控制工具可见性，不改变能力注册，不覆盖 Deadmouse Core，也不替机器或 Lead 做策略决策。

### 6. capability packages

主要代码边界：

- `src/protocol/package.ts`
- `src/protocol/manifest.ts`
- `src/protocol/manifestBundle.ts`
- `src/protocol/port.ts`
- `src/capabilities/packages/discovery.ts`
- `src/capabilities/packages/lifecycle.ts`
- `src/capabilities/registry.ts`
- `src/cli/commands/capability.ts`

capability package 是扩展能力的停靠与治理单元，不是策略单元。
每个 package 必须声明 `deadmouse.capability-port`：runner invocation、permission boundary、foreground output、artifact declarations、closeout behavior 和 wake behavior。Port 只规定生态如何进港、运行事实如何可见、产物如何登记、退场如何交卷；生态内部如何发挥由生态自己决定。
manifest package 可以通过 `.deadmouse/capabilities/*.capability.json` 安装到项目，但 manifest 必须包含完整 port。
启用且已安装的 package 会进入统一 capability registry；禁用或未安装的 package 不暴露给 Lead，但仍会被 runtime doctor 诊断。
package governance 当前包含 enabled、installed、installRef、dependencies、versionConstraints 和 diagnostics。
依赖和版本约束由机器层做硬事实检查；检查结果只说明扩展健康状态，不替 Lead 选择能力。
当前 CLI 已提供 `capability package install/list/enable/disable/doctor/test`，用于管理项目内 manifest 生命周期。

### 7. Dreaming external agent ecology

主要代码边界：

- `src/capabilities/dreaming/`
- `src/capabilities/tools/packages/dreaming/dreamingStartTool.ts`
- `src/execution/foregroundStream.ts`
- `src/runtime-ui/executionForeground.ts`

Dreaming 是当前内置的第一个完整 external agent ecology。
它通过 `external_agent.dreaming` capability package 暴露给 Lead，入口工具是 `dreaming_start`。
机器层只暴露能力、创建 assignment-backed execution、启动 runner、记录状态和前台流；是否启动 Dreaming 必须由 Lead 显式选择。

Dreaming 的运行世界分成：

- Real World：真实项目源码。
- Mirror World：Dreaming 专属 git worktree。

Dreaming 可以在 Mirror World 内自由修改、验证、迭代和产出证据。
Dreaming 默认不得修改 Real World；Real World 只有运行账本、execution stream、closeout 等机器层证据会更新。
Dreaming closeout 只产出 Mirror World 路径、执行流、验证事实、风险和 merge proposal；是否从 Mirror World 合并回 Real World 必须由用户批准。
Dreaming 运行时 Lead 挂起等待 execution closeout，终端前台切到 Dreaming execution stream，显示 Dreaming 在 Mirror World 内的实时进展。

### 8. Dreaming Loop workflow

主要代码边界：

- `src/capabilities/workflows/registry.ts`
- `src/capabilities/dreaming/loopState.ts`
- `src/capabilities/dreaming/ids.ts`
- `src/capabilities/tools/packages/dreaming/dreamingLoopTools.ts`

Dreaming Loop 是 `workflow.dreaming-loop`，不是新的 external agent。
它组织多轮 Dreaming，但不替 Lead 做继续、停止、换方向或合并判断。

Dreaming Loop 暴露三个 Lead-only 工具：

- `dreaming_loop_start`：创建带时间戳的 loop ledger，不启动 Dreaming。
- `dreaming_loop_next`：在 Lead 显式要求时启动一轮 Dreaming。
- `dreaming_loop_status`：同步并读取 loop 的事实状态。

Loop id 使用 `dreaming-loop-YYYYMMDD-HHMMSS-xxxxxx`。
Round id 使用 `dreaming-YYYYMMDD-HHMMSS-rNN-xxxxxx`。
机器层可以记录每轮 execution、foreground stream、closeout、artifact 和终态事实；不能评价本轮是否“好”、不能决定下一轮、不能自动合并 Real World。

## 文档链路示例

当前文档读取链路是扩展机制落地最完整的一组例子：

- builtin tools 暴露 `mineru_pdf_read`、`mineru_image_read`、`mineru_doc_read`、`mineru_ppt_read`
- skills 侧分别提供 `mineru-pdf-reading`、`mineru-image-reading`、`mineru-doc-reading`、`mineru-ppt-reading`
- runtime 在同一注册与治理链路里决定这些工具何时可见、何时优先于泛化的 `read_file`

这意味着 PDF、图片文档、Word 和 PPT 的读取，不是靠 prompt 记住一张散装路由表，而是靠正式扩展口把能力长进系统。

## 主路径

### builtin tool

`builtinCatalog -> registry -> governance -> runtime exposure`

### skill

`discovery / index -> explicit load_skill -> 当前 turn`

skill 只组织 workflow，不直接绕过工具边界长出动作入口。
skill 不做 turn 级自动匹配、required 推进或 missing required 阻塞。

### MCP

`configured MCP server -> discovery -> ToolRegistrySource(kind=mcp) -> runtime registry -> governance -> exposure -> invoke`

当前 MCP 只保留通用接入壳，不内置具体 MCP server，也不在 turn 启动时暴露任何默认 MCP 工具。

### host extra tool

`host tool -> ToolRegistrySource(kind=host) -> runtime registry -> governance -> exposure`

### profile

`profile registry -> selected profile -> Profile persona layer + Profile runtime facts layer -> 当前 turn`

profile 必须通过配置显式选择，当前内置 profile 包括 `intp`、`grok`、`caveman` 和 `buddha`。新增 profile 必须同时明确 persona 和 runtime facts presentation，不能把机格内容写回核心提示词主干。

### capability package

`.deadmouse/capabilities/*.capability.json -> manifest parser -> package governance -> runtime capability registry -> Lead-facing capability presentation`

package 生命周期主路径：

`local manifest with capability-port -> capability package install -> .deadmouse/capabilities/*.capability.json -> enable/disable -> doctor/test -> registry exposure`

### Dreaming

`external_agent.dreaming -> dreaming_start -> AssignmentContract -> dreaming Execution -> Mirror World runner -> foreground stream + artifacts + merge proposal -> CloseoutContract -> WakeSignal -> Lead`

### Dreaming Loop

`workflow.dreaming-loop -> dreaming_loop_start -> loop ledger -> Lead -> dreaming_loop_next -> one Dreaming Execution -> closeout/wake -> dreaming_loop_status -> Lead decides next action`

## 模块职责与非职责

### 扩展机制负责

- 给 builtin、skill、MCP 和 host extra tool 提供统一入口
- 给 profile 提供统一入口
- 给 capability package 提供可安装、可禁用、可诊断、带依赖和版本约束的治理入口
- 给 capability package 提供统一 port，让 external agent、workflow、tool、skill、MCP、Dreaming 都通过同一停靠协议进入系统
- 给 capability package 提供本地生命周期 CLI：安装、列出、启停、诊断和结构测试
- 给 Dreaming 这类 external agent ecology 提供 Capability -> Assignment -> Execution -> Progress/Artifact -> Closeout -> WakeSignal 的正式闭环
- 给 Dreaming Loop 这类 workflow 提供显式轮次账本、时间戳命名和 Lead-owned continuation
- 给扩展源打上来源和治理语义
- 把工具定义层和工具执行层分开：Lead 先看到能力，重型外部服务等真正调用时再启动
- 在 registry 边界处理冲突与 fail-closed
- 把核心提示词、profile 人格提示词和 profile 运行事实分层度量

### 扩展机制不负责

- 让 skill 直接创建平行动作入口
- 让 MCP 或宿主绕开 shared registry
- 让 profile 隐藏工具、改变权限或重写机器/模型边界
- 让 runtime facts presentation 把事实变成策略指令
- 让 package 排序、依赖或诊断结果变成机器替 Lead 选择能力
- 让 Dreaming 自动启动、自动合并 Real World、或把机器层执行能力变成策略决策
- 让 Dreaming Loop 自动判断下一轮、自动启动下一轮、或替 Lead 评价一轮成果价值
- 为了提前检查外部服务而阻塞 Lead 上场
- 用 prompt 文案代替正式扩展边界
- 让 port 变成生态内部驾驶员，限制生态自己的探索、创作、审计或自我优化路线

## 失败路径与异常路径

当前明确处理：

- 重名工具注册在边界显式失败
- 缺 governance 的扩展 fail closed
- host extra tool 会被标记为 `host`，不能伪装成 builtin
- MCP 每个已配置并成功发现的 server 形成正式 source，再进入共享注册链路
- Deadmouse 不内置具体 MCP server；新增 MCP 能力必须通过显式 server 配置或后续正式扩展接入
- 未知 profile 在边界显式失败，不能静默回退到另一个机格
- capability package 重复 id、缺失依赖、版本约束不满足、机器策略权限越界时，runtime doctor 给出结构化诊断
- capability package 缺失 port、前台输出不接入 runtime-ui、或 port runner 不完整时，manifest 解析或 runtime doctor fail closed
- disabled package 不进入 Lead 可见能力面，但不会从诊断事实里消失
- Dreaming 写路径、patch 目标和 shell cwd 必须留在 Mirror World；嵌套调度、工作树管理和后台执行工具在 Dreaming runner 内不可用
- Dreaming Loop 只能在 Lead 调用 `dreaming_loop_next` 时启动一轮 Dreaming；`dreaming_loop_status` 只同步 execution 终态事实，不做价值判断

## 当前落地决定

当前扩展机制已经固定为“工具、技能、MCP、宿主注入、profile、capability package、external agent ecology”这些正式扩展口。增加扩展必须沿这些边界接入，不再回到扁平拼接或散装 prompt 注入式扩展。
