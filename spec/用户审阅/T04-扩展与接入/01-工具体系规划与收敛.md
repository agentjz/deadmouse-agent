# 工具体系规划与收敛

## 这份文档回答什么问题

这份文档说明 Deadmouse 当前的工具体系已经收敛成什么样，以及为什么当前工具面不再继续发散。

## 当前目标与状态

当前工具专项已经完成以下收口：

- 保留现有正式工具命名
- 不增加平行工具族
- 保持正式工具默认全量可见
- 增加 `find_files`
- 收紧 `edit_file`
- 增强 `read_file`
- 增强 `search_files`
- 轻量化 `list_files`
- 增加 `code_symbols / code_references / code_pattern` 作为代码事实取证工具

当前结论已经明确：Deadmouse 借鉴外部项目时，借的是工具设计和约束，不是整套工具壳子。

同时，Capability Package 已经从单纯运行时适配层收紧为扩展治理单元：可以通过项目内 manifest 安装，可以禁用，可以诊断，可以声明依赖和版本约束。
当前还补齐了 provider harness 和 regression case：前者用于离线复现 provider 行为，后者用于把真实 session/trace 固化成可运行回归证据。

## 适用范围

当前收口覆盖的是：

- 文件探索工具
- 文件写入与精修工具
- 运行时的正式工具面
- 模型选择工具时的主要路径

这里的“收敛”指不再制造平行工具入口，不指按阶段隐藏工具，也不指让机器层替模型选择工具。

## 范围外事项

当前明确不做：

- 为同一能力再造一套平行工具名
- 把 shell 别名直接搬成正式工具名
- 把外部项目的整套工具框架搬进仓库
- 按阶段裁剪模型可见工具集合
- 让治理元数据变成机器代选工具的依据

## Capability Package 当前含义

Capability Package 现在承担的是“能力治理”，不是“自动使用”。

当前 package 可以表达：

- 是否已安装
- 是否启用
- 安装来源
- 依赖 package
- 版本约束
- 诊断信息

启用且已安装的 package 会进入 Lead 可见的能力面；禁用 package 不进入 Lead 可见能力面，但会被 `doctor runtime` 看到。
依赖缺失、版本不满足、重复 id、机器策略权限越界，都会进入结构化诊断。

这不会让机器替 Lead 选择能力。它只回答“有什么、是否健康、边界是什么”。

## 核心场景

当前工具面按场景已经清楚分工：

- 看目录，用 `list_files`
- 按路径模式找文件，用 `find_files`
- 按内容查找，用 `search_files`
- 读取文件正文，用 `read_file`
- 查代码符号、引用和结构模式，用 `code_symbols / code_references / code_pattern`
- 小范围精修已有文件，用 `edit_file`
- 需要 patch 语义时，用 `apply_patch`

其中 `run_shell` 仍然保留，但不再承担文件探索或普通读写的默认入口。
`run_shell` 当前补齐了结构化 runtime 结果：状态、超时/中断标记、输出截断标记和落盘路径。

`search_files` 当前按取证梯度提供三种输出：`files` 只返回匹配文件，`count` 只返回匹配分布，`matches` 返回匹配行和少量上下文。默认输出是 `files`，先低噪声定位，再由 Lead 决定是否继续读取。搜索结果只提供事实和后续 `read_file` 参数，不替 Lead 选择读取对象。

`code_symbols / code_references / code_pattern` 是工程级代码事实工具：它们返回声明、引用、结构模式、行号和后续 `read_file` 参数，不自动判断影响范围，不自动选择修改点，不替 Lead 产生修复策略。

## 关键约束

当前工具体系坚持以下约束：

- 同一能力只保留一条正式入口
- 写工具的约束强度高于只读工具数量
- 工具名优先表达动作和对象，不为“更短”而改名
- 工具仍然通过统一 registry、governance 和宿主注入边界接入
- 写工具统一把 `changedPaths`、diff、diagnostics、session diff 收口到正式结果
- 工具排序属于能力呈现层，只能改变展示顺序，不能减少默认可见工具集合
- 工具治理只服务执行边界、结果信号和收口证据，不替 Lead 做策略选择
- 网页工具排序不再塑造“浏览器优先”的唯一正确路线；轻量 HTTP、下载和浏览器能力都应可见，由模型按任务判断
- `load_skill` 是显式加载入口，不是机器层替模型选择路线的硬门

## 验收标准

当前专项按下面的结果验收：

- `find_files` 已成为正式的路径模式查找工具
- `edit_file` 已要求正式 identity 与锚点
- `read_file` 已具备 continuation 语义
- `search_files` 已具备 `files / count / matches` 分层取证输出
- `code_symbols / code_references / code_pattern` 已进入正式只读代码取证工具面
- `run_shell` 已具备结构化执行状态与长输出截断落盘
- 工具呈现顺序调整不会减少默认工具集合
- 工具呈现顺序调整不会把技能加载、浏览器、后台或队友通道变成默认强制路线
- closeout 的局部防打转过滤不会外溢到普通执行阶段
- 当前工具面没有长出 `read / write / edit / bash / grep / ls` 这类平行入口
- capability package 已经具备安装、禁用、诊断、依赖和版本约束的正式治理面
- scripted provider harness 已经能离线复现文本、工具调用、空响应、错误和中断
- regression case 已经能从 session/trace 捕获证据并运行回归检查

## 当前定稿

Deadmouse 当前的工具体系已经进入“少而准、边界清楚、主路径稳定”的定稿状态。重点不是复制 shell 名字，而是让正式工具更低噪声、更可恢复、更可治理，并把代码事实能力放进清晰的只读取证面。

扩展治理、provider harness 和 regression case 共同补上的是工程纪律：能力能管理，provider 行为能复现，真实失败能沉淀成回归证据。它们都不替 Lead 判断任务路线。
